## 2. Authentication, Identity & Isolation (primary section)

This is the section we drill hardest. Be concrete and reference your code.

### 2.1 Topology

Who owns session creation and validation? How does the Python API learn which authenticated
user a given request belongs to: a shared session token, a verified JWT, a shared session
table, something else? Why did you choose this over the alternatives? Draw the trust
boundary.

### 2.2 RLS Enforcement

- Which tables have RLS policies, and what does each policy say (`USING` / `WITH CHECK`)?
- What database role does the application connect as, and how does it differ from the
  migration/owner role? Why must it not be the owner or a `BYPASSRLS` role?
- How does the current user's identity reach the database session so the policies can act on
  it (for example, a `SET LOCAL` of a setting read via `current_setting(...)`)?

### 2.3 Connection Pooling Interaction

How does your identity-propagation mechanism behave under connection pooling? If a
session-scoped setting outlives a request and a pooled connection is reused by another user,
isolation leaks. Explain exactly why that does not happen in your design.

### 2.4 The Worker's Identity

The background worker runs outside any HTTP request and has no session. Under whose database
identity does it read a pending job and write the result? How do you stop the worker from
becoming a hole that bypasses the isolation every other path enforces?

### 2.5 The Isolation Guarantee

State, in one or two sentences, why a request from user A can never read user B's data, even
when some application-layer `WHERE` clause is wrong or missing. This is the claim we will try
to break.

## 3. Agent Design

### 3.1 Tools

List every tool you gave the agent. For each one: what it does, why you included it, and what
it returns and in what format. If you considered tools you decided against, explain why.

### 3.2 Navigation Strategy

How does your agent approach a document it has never seen? How does it figure out what kind
of document it is, where the relevant data is, and what order to extract in? How does it
handle a table that splits across pages?

### 3.3 State Management

What state does the agent maintain across tool calls? Where does it live, how is it
structured, and why that structure?

### 3.4 Uncertainty and Flagging

How does your agent decide when to output a record vs. flag it for review? What signals does
it use? How do you avoid both confident-wrong rows and flagging-everything?

## 4. Job Queue & Reliability

### 4.1 Worker and Concurrency

Two workers run concurrently. Walk through the exact mechanism your worker uses to claim
jobs: what query runs, how you guarantee two workers never claim the same job, and what gets
written back and when. What would break if you removed the safeguard?

### 4.2 Crash Recovery

What happens if the worker dies mid-job? How and when does the system recover the job?

### 4.3 Status Transitions

Describe every status transition, what triggers each, and where in the code it happens.

### 4.4 Retry Policy

How many retries, what backoff, and which errors are retryable vs. fatal? How did you test
it?

### 4.5 Cost and Latency Tracking

How do you calculate and store `cost_usd`, `token_usage`, and `processing_duration_seconds`?
Where in the code, and what pricing assumptions? There are no hard targets this round, but
show how you measure.

### 4.6 Result Caching

How do you detect that an upload matches a previously completed extraction, and how does the
cache respect the per-user isolation boundary so that a cache hit never crosses users? Where
is the fingerprint stored, what does a cache hit return, and what are the failure modes? What
does the bypass flag do at the implementation level?

## 9. Extending the Design (prep for the debrief)

You will be asked to extend your own system live. Jot your initial thinking here; we will go
deeper on the call.

- Tenant-level isolation: how would you move from per-user to per-organization (tenant)
  isolation? What changes in the schema, the policies, and the identity plumbing?
- Cross-tenant sharing: how would you let one tenant share a specific document with another,
  enforced at the RLS level, without weakening the default-deny boundary?
- Multi-worker / workflow: how would you turn the single-queue worker into a multi-worker,
  workflow-based system across machines and regions, and how would the user/tenant identity
  travel to a detached worker there?
